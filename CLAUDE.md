# Career_Coach — LLM Operating Manual

*Read this first every session. This wiki is a personal career intelligence platform for its owner: job-market intelligence + career coaching. Markdown is the database; you own the prose, scripts own the numbers.*

> **First run (new instance):** If `USER.md` still has an empty `name:`, this is a freshly-cloned, unconfigured instance. Greet the user and tell them to run **`/setup`** — the onboarding command that installs dependencies, reads their résumé, calibrates `USER.md`, builds and launches the first dashboard (teaching the basics), and offers a private backup. Do **not** improvise onboarding or hand-write `USER.md` / `coach/Profile.md` yourself — `/setup` is the structured flow. (It's user-invoked, so you can't auto-run it; just have them type `/setup`.)

> **Maintaining the engine?** If this repo has a `docs/superpowers/` folder (the template's own dev repo), start there — `HANDOFF.md` for build status and `MAINTAINING.md` for the publish/update runbook. Cloned instances won't have it; this manual covers your wiki content workflows.

## The user (calibrates everything)

All per-user calibration lives in **`USER.md`** (repo root): `name`, target `tracks`, `geo-zones` (+ match scores), `level-band`, `employment-types`, `exclude` (never-ingest domains), `pivot`, `resume-version`. **Read `USER.md` at the start of every session** — it is the source of truth for who this instance serves. The deterministic indexer reads `tracks` (fit track-score) and `geo-zones` (fit geo-score) from it. It also reads `comp-floor` (below-floor flag/penalty), `clearance`/`work-auth` (an inferred clearance-realism penalty), and `pivot` (a gap-ROI boost toward the `into:` tracks) — each dormant until that field is set.

The indexer reads this instance's calibration from `USER.md` — tracks, geo-zones, level band, employment types, exclusions, pivot, and the optional comp/clearance signals. Nothing about the user is hardcoded here; read `USER.md` to see who this instance serves.

**Tracks are user-defined labels** (any field — e.g. `backend`, `icu`, `brand`, `robotics`); the engine treats them generically. The track/skill examples and the domain rules throughout this manual (notably the ROS and clearance rules in the Ingest workflow, and skill examples like `fpga-tools`/`can-bus`) are **illustrative for an engineering user** — adapt them to the user's actual field.

## Ownership rules (hard lines)

- `raw-sources/` is **immutable**. Resumes are versioned (`Resume YYYY-MM.pdf`), never replaced.
- `analytics/` and `data/` are **indexer-owned** (Phase 3+). Never hand-edit.
- Derived values (fit scores, demand tiers, momentum, freshness confidence) are **never written into authored frontmatter** — they live in `data/`.
- Fit scoring and gap ROI are indexer-owned (`tools/indexer/src/fit.ts`, `gap.ts`). Tunable knobs live in `tools/config.json`: `fitWeights`, `fitTiers`, `fitTuning` (tier weights / shrink k / depth bonus), and `relatedHave` (partial-credit map, e.g. ROS 1 → ROS 2). Re-run `npm run index` after changing them. Per-user geo-zone scores and target tracks now live in **`USER.md`**, not `config.json` (which holds only universal scoring defaults).
- The server (**live since Phase 4a** — `npm run coach`, :4280) writes only: `app-status`, `app-updated`, `app-history`, `rejection-stage`, `rejection-reason`, and `## Notes` appends on job pages — via two POST endpoints, targeted line edits, atomic writes.
- Every new page → add to `index.md`. Every operation → one line in `log.md`.
- `data/` is git-ignored and regenerated by `npm run index`; a fresh clone needs one index pass before the dashboard serves. `template/` and `examples/` are template-packaging artifacts — the indexer's globs are root-rooted and never scan them.
- `assets/logos/` holds downloaded company logos — git-tracked locally (so they back up and sync) but **never shipped** to the public template (excluded from the export manifest like `jobs/`, `companies/`, etc.).

## Conventions

- Filenames: Title Case with spaces. Job pages: `<Company> — <Title>.md` (em dash), disambiguate with city or date if needed.
- Internal links `[[Page Name]]`; external links standard markdown. Link liberally; a link to a not-yet-written page marks a gap.
- All wiki pages carry frontmatter with at least `type`, `created`, `updated`.
- The user-facing command layer lives in `.claude/skills/<name>/SKILL.md` (prompt templates calibrated by `USER.md`). `/setup` onboards a new instance; `/find-jobs`, `/add-job`, `/coach`, `/update-resume`, and `/lint` wrap the documented workflows (Ingest batch / Manual drop / Coaching session / Profile update / Lint); `/launch`, `/status`, `/backup`, `/help`, `/watch`, `/enrich`, and `/update` round out the operational set (`/launch` starts the local dashboard server at :4280); and `/why`, `/compare`, and `/market-trends` are decision tools over the indexed fit data and the broader market. `/enrich` backfills richer job/company info (verbatim postings, company description/HQ/founded/domain/logo) on existing pages.

## Job page schema (`jobs/`)

```yaml
---
type: job
title: Robotics Software Engineer
company: Boston Dynamics            # must match a companies/ page name
location: Waltham, MA
address: "100 Beaver St, Waltham, MA 02453"   # optional; verbatim when shown
geo: boston-metro                   # must match a USER.md zone slug, or remote | other
track: [robotics, software]         # 1+ of the user's USER.md `tracks` (user-defined labels; values here are just an example)
level: early                        # entry | early | mid | senior | unknown
salary: "$95k–$120k"                # optional; verbatim
deadline: 2026-07-15                 # optional; application deadline when the posting shows one
url: https://...                    # CANONICAL posting URL only (dedup key); leave "" if unresolved
discovered-via: https://...         # aggregator/list URL when url is empty (never used for dedup)
employment: full-time               # full-time (default, may omit) | co-op | internship | contract
source: batch-2026-06-10            # batch-YYYY-MM-DD | manual
posted: 2026-06-01                  # omit if unknown
ingested: 2026-06-10
status: active                      # active | stale | closed
app-status: none                    # none | interested | applied | interview | offer | rejected
app-updated: 2026-06-10
app-history: []                     # server-appended "YYYY-MM-DD status" strings
rejection-stage: screening          # only when rejected: screening | oa | phone | onsite | offer-declined
rejection-reason: ""                # optional free text; only with rejected
skills: [ros2, cpp, python]         # canonical slugs; each must resolve to a skills/ page
keywords: [perception]              # softer terms, counted at lower weight
created: 2026-06-10
updated: 2026-06-10
---
```

Body sections, in order: one-line gist (`>` quote), `## Summary` (2–4 sentences, synthesized — never paste the posting), `## Requirements` (distilled bullets), `## Fit notes` (REQUIRED, non-empty: how this matches the user specifically), `## Red flags` (ghost-job signals: reposts, very old posting, vague scope), `## Notes`, and an optional final `## Posting (verbatim)` — the exact posting wording lifted from the listing (role/responsibilities/requirements/about), with EEO/legal/benefits boilerplate trimmed. `## Summary` stays synthesized (never paste the posting there); verbatim text lives **only** in `## Posting (verbatim)`.

## Skill page schema (`skills/`)

```yaml
---
type: skill
aliases: [ROS2, ROS 2, Robot Operating System]
slug: ros2                          # canonical slug used in job frontmatter
category: robotics-software         # authored domain grouping (NOT demand tier)
have: false                         # does the user have this skill?
evidence: ""                        # if have: where demonstrated (resume / GitHub / project)
created: 2026-06-10
updated: 2026-06-10
---
```

Body: one-line definition + short synthesis (what it is, where it shows up, how it's learned/demonstrated). One page per skill; aliases must be unique across all skill pages.

## Company page schema (`companies/`)

```yaml
---
type: company
hq: Waltham, MA
industry: robotics
size: ~1100                          # optional; employee count / band, verbatim
founded: 1981                        # optional; founding year
domain: cognex.com                   # optional; bare domain — used for the website link + logo fallback
careers-url: https://...
created: 2026-06-10
updated: 2026-06-10
---
```

Body sections: one-liner, `## About` (a short **verbatim, sourced** company description — keep it distinct from your synthesized `## What they do`), `## What they do`, `## Openings observed` (wikilinks to job pages + status), `## Contacts` (lightweight CRM: name — role — context), `## Your history`.

**Logo:** store the company's logo at `assets/logos/<company-slug>.<ext>` (png/svg/jpg/jpeg/webp), where `<company-slug>` is the page name lowercased with each run of non-alphanumeric characters (spaces, punctuation, em-dashes) replaced by a single hyphen and leading/trailing hyphens trimmed (e.g. `Pickle Robot` → `pickle-robot`, `H2O.ai` → `h2o-ai`). The indexer resolves it into the dashboard; absent → the two-letter monogram is shown.

## Coach pages (`coach/`)

- `Profile.md` (`type: profile`; frontmatter also `resume-version`, `resume-date`) — experience summary, skills-with-evidence, positioning statement. Skill possession mirrors into `skills/*.md` `have:`/`evidence:`.
- `Goals.md` (`type: coach`) — living document; refine during coaching sessions.
- `Skill Gap Analysis.md` (`type: coach`) — indexer writes between `<!-- gap:begin -->`/`<!-- gap:end -->`; LLM commentary lives below the markers and must survive regeneration.
- `Portfolio Roadmap.md` (`type: coach`) — strategy overview linking `projects/` pages.
- `projects/<Project>.md` (`type: project`; `status: idea|in-progress|done`, `closes: [slugs]`, `repo:`) — one page per portfolio project.
- `briefs/<YYYY-MM-DD> Batch Brief.md` (`type: brief`) — see Ingest workflow step 7.
- `Market Trends.md` (`type: market-trends`; top-level `researched:` + `updated:` + global `sources:`) — broad, externally-sourced market research per `USER.md` track, authored by `/market-trends`. A `market:` frontmatter block holds a `tracks:` list, one entry per track: `track` (a `USER.md` track label — required; the scorecard's "your fit" derivation matches on it), `trajectory` (rising|steady|cooling), `headline`, `demand` (1–5), `growthNote` (sourced number, optional), `competition` (low|moderate|high|very-high), `salary` (by level), `coreSkills`/`emergingSkills`/`fadingSkills` (exact skill slugs), `tailwinds`/`headwinds`/`hiring` (+ optional `hiringCompanies`), `snapshot` (sourced prose with `[n]` markers), `yourFit`, and optional per-track `sources`. Consumed by `/api/market` → the dashboard **Market Trends** view. A `## Notes` body section survives regeneration. Re-verify (re-run `/market-trends`) when older than 6 months.
- `interviews/` (`type: interview-bank`) — grows organically; entries track question, where seen, frequency, difficulty, confidence.
- `advice/<Topic>.md` (`type: advice`; `researched: YYYY-MM-DD`, `sources: [urls]`) — re-verify anything older than 6 months.

## Workflow: Ingest batch

1. **Harvest first (cheap sourcing):** run `npm run harvest` in `tools/` → `data/board-cache.json`, a dedup-tagged list of **currently-live** postings pulled from the Greenhouse/Lever/Ashby JSON board APIs of every tracked company + `find-jobs/watchlist.json`, filtered to this user. Author the bulk straight from the `"known": false` entries (live by construction, `text` included — no web search or liveness check needed). THEN web-search only for net-new companies not in `data/known.json` `companies`, and screen every discovery candidate with the one-line helpers before any WebFetch: `node tools/known.mjs "<url-or-Company|Title>"` (dupe) and `node tools/checkurl.mjs "<url>"` (liveness — `live` vs dead/`redirected` soft-404). Then web-search across tracks × geographies: company career pages, public ATS boards (Greenhouse/Lever/Ashby), aggregators. **Real postings only** — capture the employer ATS URL; never invent. **Confirm the URL is live:** WebFetch the canonical posting and verify it returns a real, specific job posting (not a 404/410, not a generic careers/search board) before including it. Skip roles matching the user's `exclude` list (`USER.md`). Skip postings where you cannot determine actual responsibilities. Target the user's level band (`USER.md` `level-band`; typically entry/early/new-grad/co-op/internship + some mid; not senior/staff/principal).
2. **Dedup** against `jobs/`: exact `url`, then normalized company+title. A repost of an existing role → note on the existing page's `## Red flags`, don't re-ingest. A *different* role at an existing company → ingest it and append to that company's `## Openings observed` (don't recreate the company page). Err toward ingesting when unsure.
3. Per new posting: job page (schema above, incl. `address:` when shown); create/update company page; create missing skill pages. **URL rule:** `url:` must be the posting's own page (employer ATS or posting-specific aggregator link); if only a generic list/search URL is known, set `url: ""` and record the list in `discovered-via:` — shared list URLs must never become dedup keys. Use `url: ""` + `discovered-via:` only when no canonical posting URL exists AND the `discovered-via` link lands on a real listing — never as a shortcut for an unverified or 404'd posting (a dead `url:` is worse than an honest empty one; the dashboard falls back to `discovered-via` gracefully). **Geo rule:** add the city to `geo/places.json` (repo root) for any non-remote zone defined in `USER.md`; `remote` and `other` skip the lookup and the map. Map bounds auto-fit to the plotted cities (no fixed region box). **Slug rule:** `skills:` must use each skill page's exact `slug:` field, which is NOT always the title (`grep '^slug:' skills/*.md` first — easy-to-miss ones: `fpga-tools`, `rest-api`, `sdr`, `cpp`, `can-bus`, `nodejs`). Put canonical, demand-signal skills in `skills:`; tooling/soft terms (kafka, terraform, numpy, agile, rtl…) in `keywords:`. A new skill page must NOT reuse an alias another skill already claims (the indexer flags "alias collision"). **ROS rule:** ROS 1 (Noetic) is EOL since 2025-05-31 and ROS 2 is the market baseline. Tag `ros2` when a posting names ROS 2 / ROS2 / Nav2 / MoveIt2 / DDS / colcon / ament, AND for a *generic* "ROS" mention with no version — UNLESS it explicitly says ROS 1 / Noetic / Melodic / Kinetic (then `ros`), or lists both (then `ros, ros2`). If the user has a related earlier skill (e.g. ROS 1), the indexer's `relatedHave` map (`tools/config.json`) credits it partially toward `ros2` while keeping `ros2` a real gap. **Clearance rule:** when a posting requires a security clearance or US citizenship/ITAR, state it explicitly in `## Requirements` or `## Red flags` (e.g. "requires an active Secret clearance") — the indexer infers a clearance-realism penalty from the body text + a defense-company list (`clearanceSignals` in `tools/config.json`), with no `clearance:` frontmatter field. **Filename rule:** filenames can't contain `/` — substitute consistently (e.g. `AI_ML`) and make every `[[wikilink]]` to that page match the filename exactly.
4. Update `index.md` (new batch section by track + companies + skills); append batch entry to `log.md`.
5. **Validation gate — run the indexer:** `npm run index` in `tools/` (regenerates `data/` + `analytics/` + the gap table). Then check `data/meta.json` `warnings` and `data/errors.json`; resolve unresolvable slugs, alias collisions, and empty `## Fit notes` before reporting (a salary-sanity warning on a legitimately high-paying senior role is expected and can be left). **Link-liveness gate:** also run `npm run checklinks` (HTTP-checks every job `url:` → `data/links.json`); resolve any `dead`/`redirected`/`none` result before committing — find the canonical URL, or set `status: stale` + a dated `## Red flags` note if the posting is genuinely gone. `checklinks` follows redirects and flags soft-404s (a closed Greenhouse listing returns 200 but redirects its specific URL to a `?error=true` board) as `redirected`, not `live`. (Known limits: `unverified` = bot-blocking ATS like Lever/Workday/some careers sites returning 403/500/timeouts for *live* roles — spot-check with a browser or `curl` before marking stale; and many ATS are JavaScript-rendered — Ashby/Workday/Lever/Gem — so WebFetch can't read the posting body even when the role is live. Judge liveness by the *final* URL after redirects, never by fetched content, and never stale an HTTP-200 same-URL role.) `npm run coach` serves the live dashboard and hot-reloads via its file watcher.
6. Report to the user: N new, M dupes skipped, top skills this batch, best fits, anything notable.
7. Write `coach/briefs/<date> Batch Brief.md`: market movements, the user's position, pipeline summary (incl. postmortem patterns), 3–5 concrete recommendations. Pull the numbers from `data/overview.json` + `data/gap.json` + `data/jobs.json` so the read is accurate. The page may carry an optional `brief:` frontmatter block (authored with the Write tool) consumed by `/api/brief`; fields: `kicker`, `headline`, `byline`, `lead` (list of 1–2 paragraph strings), `pullStat {number, label}`, `threeRoles` (list of `{job: <indexer-slug-id>, why}` pairs — get ids from `data/jobs.json`), and `nudge`. The most-recently-dated brief file wins; omitting the block falls back to the templated brief.

**Scaling a big batch (≈10+ postings):** parallelize with subagents — one *research* agent per track (real ATS postings only → structured data: company/title/geo/track/level/url/salary/skills/keywords/responsibilities/red-flags), then one *authoring* agent per track for its job + company pages. Keep the shared work central to avoid file collisions: YOU create the new skill pages, edit `geo/places.json`, `index.md`, `log.md`, and write the brief; hand each authoring agent the exact canonical slug list and the job/company templates. Tracks partition cleanly (a company rarely spans two tracks), so per-track authoring agents don't collide. Run the research/authoring agents on a cheaper model (**Sonnet**) — the work is mechanical fact-find + quote-lift + page edits, and Opus burns the session limit several-fold faster; keep waves small (≈3 agents) so each flushes its writes before any limit cutoff, and have agents append durable sections (`## Posting (verbatim)`, `## About`) *before* bumping `updated:` so a cutoff leaves each file fully-done-or-untouched.

## Workflow: Manual drop

The user pastes a URL or posting text → preserve raw text under `raw-sources/postings/` first, then steps 2–6 above for that one posting (brief optional).

## Workflow: Coaching session

Read `coach/` + relevant `analytics/` + job pages first; answer with citations to wiki pages; fold durable insight back into `Goals.md` / `Portfolio Roadmap.md` / project / advice pages; log the session.

## Workflow: Profile update

New resume → versioned into `raw-sources/`, re-synthesize `Profile.md` (bump `resume-version`/`resume-date`), summarize what changed vs. the prior version, reconcile `skills/*.md` `have:`/`evidence:`, log the delta.

## Workflow: Lint

LLM lint: contradictions; advice/Market-Trends `researched:` > 6 months; orphan pages; missing cross-links; duplicate company name variants; dead-link spot checks; `have:` flags contradicting Profile; mentioned-but-unwritten pages. Findings → `log.md` under `lint YYYY-MM-DD`. (Deterministic checks — salary sanity, alias collisions, empty Fit notes, unresolvable slugs, and company-enrichment gaps (a company page missing a logo file or `## About`; confidential/undisclosed employers exempted) — are the indexer's job from Phase 3, surfaced in `data/meta.json` `warnings`.)

## Per-page ingest checklist (run mentally before saving any job page)

- [ ] All frontmatter enums valid; `geo` and `track` from the fixed vocabularies
- [ ] `url:` is canonical-or-empty AND WebFetch-confirmed live (generic list URLs go in `discovered-via:`; never leave a dead/404 `url:`)
- [ ] Non-full-time postings carry `employment:` (co-op/internship/contract)
- [ ] Every `skills:` entry is an exact `slug:` value (not a page title); soft/tooling terms in `keywords:`; new skill pages don't reuse another skill's alias
- [ ] `## Fit notes` non-empty and specific to the user
- [ ] Company page exists and links back (existing company → append to `## Openings observed`, don't recreate) — with `domain`, `founded`, HQ, `## About`, and a logo file at `assets/logos/<slug>.<ext>` when available (run `/enrich` to backfill).
- [ ] Filename has no `/`; every `[[wikilink]]` to the page matches the filename exactly
- [ ] City present in `geo/places.json` (repo root) for any non-remote `USER.md` zone
- [ ] `posted:` date captured whenever the posting shows it — the momentum series buckets by posted/ingested date, and missing posted dates collapse the whole corpus into one bucket
- [ ] Verbatim posting captured into `## Posting (verbatim)` (boilerplate trimmed); `deadline:` set when the posting shows one
- [ ] `index.md` + `log.md` updated; `npm run index` clean (no new warnings/errors); `npm run checklinks` clean for this page (link live, or `status: stale` + dated red flag if gone)
